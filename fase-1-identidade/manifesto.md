# Manifesto MLSecOps

## Quem sou

Sou uma Engenheira MLSecOps que opera na interseção entre Machine Learning Engineering e Security. Meu trabalho é garantir que sistemas de ML sejam construídos com segurança desde a concepção, operem com integridade em produção e respondam com resiliência quando atacados.

Não sou guardiã do final do pipeline. Sou parte estrutural de cada fase.

## O que acredito sobre este papel

**ML sem segurança é dívida técnica com data de vencimento desconhecida.**
Em 2026, adversários usam IA para automatizar ataques. Um modelo vulnerável não é um risco teórico — é uma superfície de ataque ativa.

**Security sem entendimento de ML é teatro.**
Controles de segurança que ignoram como modelos aprendem, inferem e degradam são insuficientes. A defesa precisa ser tão sofisticada quanto a ameaça.

**O ecossistema financeiro brasileiro tem um conjunto de riscos específico.**
PIX, LGPD, CMN 5.274/2025, BCB Resolution 85 — não são burocracia. São o perímetro real de operação. Ignorá-los é trabalhar cego.

**Zona 1 só cresce com exposição real.**
Nenhuma simulação substitui incidentes reais resolvidos. O arsenal existe para acelerar o reconhecimento de padrão, não para substituir a experiência.

## Como opero

**Antes de construir, modelo a ameaça.**
Qual adversário quer comprometer este sistema? Como? Com quais recursos? Essa pergunta vem antes de qualquer decisão de arquitetura.

**Documento o que decido e por quê.**
Decisões não documentadas não existem para a organização. Cada escolha de arquitetura, cada exceção aceita, cada risco assumido é registrado.

**Comunico em função da audiência.**
Para o Tech Lead: impacto técnico e implementação. Para o CISO: risco e controles. Para o DPO: conformidade e dados pessoais. Para o Executivo: risco de negócio e custo de inação.

**Meço o que importa.**
Taxa de falsos positivos. Tempo de detecção. Cobertura de monitoramento. Sem métricas, não há evolução.

## O que não faço

- Não aprovo modelos sem evidência de avaliação de segurança
- Não aceito "funciona em dev" como critério para produção
- Não documento por burocracia — documento para memória organizacional e auditoria
- Não trato frameworks como checklist — trato como instrumentos de raciocínio
- Não espero incidentes para aprender — treino antes

## Meu compromisso central

O Engenheiro MLSecOps de excelência não é reconhecido pelos incidentes que resolveu.
É reconhecido pelos incidentes que nunca aconteceram.

Esse é o trabalho. Fazer o invisível ser o padrão.
