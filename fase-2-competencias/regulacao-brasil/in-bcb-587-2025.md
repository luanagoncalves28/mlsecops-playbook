# Instrucao Normativa BCB 587/2025

## O que e

Instrucao Normativa do Banco Central do Brasil que complementa e detalha os procedimentos operacionais da Resolucao BCB 538/2025, especificando como as instituicoes devem implementar os controles de seguranca cibernetica, incluindo disposicoes especificas para sistemas baseados em IA e ML.

**Tipo**: Instrucao Normativa (IN) - carater operacional e procedimental
**Relacao**: Complementa a Resolucao BCB 538/2025

---

## Principais Disposicoes

### Classificacao de Sistemas Criticos
A IN 587 estabelece criterios para classificar sistemas de TI/IA em niveis de criticidade:

| Nivel | Criterio | Requisitos |
|-------|----------|------------|
| Critico | Afeta servicos essenciais, >10k clientes impactados | Monitoramento 24/7, RTO < 4h |
| Alto | Dados sensiveis, transacoes financeiras | Monitoramento continuo, RTO < 8h |
| Medio | Processos internos, dados nao sensiveis | Monitoramento diario, RTO < 24h |
| Baixo | Sistemas auxiliares | Monitoramento periodico |

**Para sistemas de ML:** modelos de deteccao de fraude, score de credito = Critico

---

## Procedimentos de Gestao de Incidentes

### Fluxo de Tratamento
```
Deteccao -> Triagem (1h) -> Classificacao (2h) -> Resposta -> Notificacao BCB -> Resolucao -> Post-mortem
```

### SLAs por Severidade

| Severidade | Tempo de Deteccao | Tempo de Resposta | Notificacao BCB |
|------------|------------------|-------------------|------------------|
| P1 - Critico | < 15 min | < 1 hora | Imediato (< 1h) |
| P2 - Alto | < 30 min | < 4 horas | < 24 horas |
| P3 - Medio | < 2 horas | < 24 horas | Nao requerida |
| P4 - Baixo | < 24 horas | < 72 horas | Nao requerida |

### Incidentes Especificos de ML
- **Data Poisoning detectado**: P1 se em producao, P2 se em staging
- **Model performance degradation**: P2 se acima de threshold definido
- **Adversarial attack detectado**: P1 se comprometimento confirmado
- **Exfiltracao de modelo**: P1 - notificacao imediata ao BCB

---

## Requisitos de Documentacao e Auditoria

### Para Sistemas de ML em Producao
- [ ] Model card atualizado (trimestral ou quando houver mudanca significativa)
- [ ] Log de decisoes automatizadas criticas (retencao 5 anos)
- [ ] Registro de incidentes com acoes corretivas
- [ ] Relatorio de testes de robustez adversarial (anual)
- [ ] Evidencia de monitoramento continuo
- [ ] Historico de versoes e deploys

### Trilha de Auditoria Requerida
Para cada predicao critica (credito, fraude, compliance):
- Timestamp
- Features de entrada (anonimizadas conforme LGPD)
- Predicao e score de confianca
- Versao do modelo utilizado
- Identificacao do request (para rastreabilidade)

---

## Requisitos de Testes

### Frequencia de Testes
| Tipo de Teste | Frequencia | Responsavel |
|---------------|------------|-------------|
| Teste de performance | Continuo (monitoramento) | MLOps Team |
| Teste de integridade do modelo | Mensal | MLSecOps Team |
| Teste adversarial | Semestral | Security Team |
| Penetration test do endpoint | Anual | Red Team |
| Teste de recuperacao (DR) | Anual | DevOps/MLOps |

---

## Gestao de Fornecedores de ML/IA

### Due Diligence Requerida
- Avaliacao de seguranca antes de contratar fornecedor de modelos
- Clausulas contratuais especificas para IA (responsabilidade, auditabilidade)
- Direito de auditoria dos sistemas do fornecedor
- Plano de saida sem dependencia excessiva

### Para Uso de Modelos de IA de Terceiros (APIs, pre-trained)
- [ ] Validar procedencia e reputacao do fornecedor
- [ ] Testar comportamento adversarial
- [ ] Verificar politica de dados do fornecedor (LGPD compliance)
- [ ] Definir monitoramento de mudancas de versao do modelo externo
